# 4. Elixir — Procesos con estado (aka servers)

Clase 4 — Del dato inmutable al proceso que administra una abstracción

## Objetivo

Integrar lo visto en las clases anteriores:

* structs y modelado de dominio
* pattern matching y actualización inmutable
* módulos como API de una abstracción
* procesos, mailbox, `send` y `receive`

La idea de hoy es mostrar cómo un proceso puede administrar **estado con forma**, no solo valores sueltos.

---

## Recap

Hasta ahora vimos dos cosas por separado:

1. cómo modelar datos y transformarlos en forma pura
2. cómo crear procesos y comunicar procesos por mensajes

Hoy juntamos ambas:

> un proceso puede mantener como estado una estructura de dominio, y cada mensaje puede transformar esa estructura devolviendo una nueva versión.

---

## Cambio de ejemplo

En lugar de usar carrito de compra, vamos a usar un modelo más simple:

# Biblioteca

Vamos a modelar una biblioteca con préstamos.

### Entidades

* `Book`

  * `id`
  * `title`
  * `author`
  * `available`

* `Library`

  * `next_id`
  * `books`

### Operaciones

Queremos poder:

* crear una biblioteca vacía
* agregar un libro
* sacar un libro
* prestar un libro
* devolver un libro
* consultar un libro
* listar disponibles

Este ejemplo sirve porque:

* usa structs
* tiene operaciones claras
* permite actualizar estado inmutable
* el proceso después se entiende fácil
* es mucho más simple que carrito + checkout

## Primera parte: abstracción pura con módulos

## Structs del dominio

```elixir
defmodule Book do
  defstruct [:id, :title, :author, available: true]
end

defmodule Library do
  defstruct next_id: 1, books: %{}
end
```

---

## API del módulo

```elixir
defmodule Library do
  defstruct next_id: 1, books: %{}

  def new do
    %Library{}
  end
end
```

La biblioteca es un dato.

Las operaciones viven en el módulo.

---

## Agregar un libro

```elixir
def add_book(%Library{} = library, attrs) do
  book = %Book{
    id: library.next_id,
    title: attrs.title,
    author: attrs.author,
    available: true
  }

  %Library{
    library
    | next_id: library.next_id + 1,
      books: Map.put(library.books, book.id, book)
  }
end
```

### Conceptos que aparecen

* struct como representación del dominio
* pattern matching en argumentos
* `Map.put/3`
* actualización inmutable
* generación de IDs

---

## Consultar un libro

```elixir
def get_book(%Library{} = library, id) do
  Map.get(library.books, id)
end
```

---

## Sacar un libro

```elixir
def remove_book(%Library{} = library, id) do
  %Library{library | books: Map.delete(library.books, id)}
end
```

---

## Prestar un libro

```elixir
def lend_book(%Library{} = library, id) do
  case Map.get(library.books, id) do
    nil ->
      {:error, :not_found}

    %Book{available: false} ->
      {:error, :already_lent}

    %Book{} = book ->
      updated_book = %Book{book | available: false}
      new_library = %Library{library | books: Map.put(library.books, id, updated_book)}
      {:ok, new_library}
  end
end
```

### Conceptos que aparecen

* `case`
* pattern matching sobre structs
* branches de error y éxito
* transformación de una estructura anidada

---

## Devolver un libro

```elixir
def return_book(%Library{} = library, id) do
  case Map.get(library.books, id) do
    nil ->
      {:error, :not_found}

    %Book{available: true} ->
      {:error, :already_available}

    %Book{} = book ->
      updated_book = %Book{book | available: true}
      new_library = %Library{library | books: Map.put(library.books, id, updated_book)}
      {:ok, new_library}
  end
end
```

---

## Listar disponibles

```elixir
def available_books(%Library{} = library) do
  library.books
  |> Map.values()
  |> Enum.filter(fn %Book{available: available} -> available end)
end
```

Acá volvemos a conectar con:

* `Map.values/1`
* `Enum.filter/2`
* pattern matching dentro de funciones anónimas
* pipeline

## Qué logramos en esta primera parte

Ya tenemos una abstracción real:

* los datos tienen forma
* la API es clara
* todas las transformaciones son inmutables
* las operaciones devuelven nuevas versiones del estado

Pero ojo con algo importante:

### Problema conceptual

Este modelo funciona perfecto en forma pura, pero no define **quién es el dueño del estado**.

Si dos partes del sistema hacen esto en paralelo:

```elixir
library1 = Library.lend_book(library, 2)
library2 = Library.lend_book(library, 2)
```

Ambas operaciones parten del mismo estado inicial.

Esto abre la puerta a inconsistencias.

No porque el código esté mal, sino porque:

> no hay coordinación sobre quién tiene la versión válida del estado.

Este es el problema que resuelven los procesos.

---

Ya tenemos una abstracción real:

* los datos tienen forma
* la API es clara

Pero todavía hay un problema:

si dos clientes usan la misma biblioteca al mismo tiempo, ¿quién decide cuál es la versión actual?

## Segunda parte: mover la biblioteca a un proceso

Ahora queremos que un proceso sea el dueño del estado.

Ese estado no va a ser un entero ni una lista suelta.

## Esqueleto del proceso

```elixir
defmodule LibraryServer do

  end

  def loop(library) do
    receive do
      _message ->
        loop(library)
    end
  end
end
```

---

## Idea clave de hoy

```elixir
}
```

No estamos guardando “variables sueltas”.

Estamos guardando una abstracción entera.

## Versión más completa del servidor

```elixir
defmodule LibraryServer do
  def start do
    spawn(fn -> loop(Library.new()) end)
  end

  def loop(library) do
    receive do
      {:add_book, attrs} ->
        new_library = Library.add_book(library, attrs)
        loop(new_library)

      {:get_book, id, caller} ->
        send(caller, {:book, Library.get_book(library, id)})
        loop(library)

      {:lend_book, id, caller} ->
        case Library.lend_book(library, id) do
          {:ok, new_library} ->
            send(caller, :ok)
            loop(new_library)

          {:error, reason} ->
            send(caller, {:error, reason})
            loop(library)
        end

      {:return_book, id, caller} ->
        case Library.return_book(library, id) do
          {:ok, new_library} ->
            send(caller, :ok)
            loop(new_library)

          {:error, reason} ->
            send(caller, {:error, reason})
            loop(library)
        end

      {:remove_book, id} ->
        new_library = Library.remove_book(library, id)
        loop(new_library)
    end
  end
end
```

## Qué queremos mostrar con este ejemplo

### 1. El proceso mantiene una estructura real

No solo un contador.

### 2. Cada mensaje puede:

* consultar
* agregar
* sacar
* transformar

### 3. El loop recursivo mantiene viva la abstracción

### 4. El estado sigue siendo inmutable

Cada operación construye una nueva versión de `%Library{}`.

## Cómo modelamos los préstamos

En esta versión de la clase, los préstamos no son un `struct` separado.

Se modelan como una relación simple dentro de `Library`:

```elixir
%Library{
  next_book_id: 3,
  next_user_id: 2,
  books: %{
    1 => %Book{id: 1, title: "Dune", author: "Frank Herbert", available: true},
    2 => %Book{id: 2, title: "Foundation", author: "Isaac Asimov", available: false}
  },
  users: %{
    1 => %User{id: 1, name: "Ana"}
  },
  loans: %{
    2 => 1
  }
}
```

La convención es:

* clave = `book_id`
* valor = `user_id`

Entonces:

```elixir
2 => 1
```

significa:

* el libro `2`
* está prestado al usuario `1`

---

## ¿Por qué modelar `loans` así?

Porque para esta clase queremos priorizar:

* claridad del modelo
* pocas entidades
* una única fuente de verdad
* una transición limpia a procesos

Este diseño permite responder fácilmente preguntas como:

* ¿el libro está prestado?
* ¿quién lo tiene?
* ¿puede prestarse ahora?

Sin agregar todavía una cuarta entidad como `%Loan{}`.

---

## Invariantes del dominio

Nuestro modelo quiere preservar estas reglas:

1. un libro solo puede estar prestado a un usuario a la vez
2. si un libro está en `loans`, no está disponible
3. si un libro no está en `loans`, puede estar disponible
4. pedir prestado y devolver deben mantener consistente la relación entre `books` y `loans`

La abstracción del módulo y luego el proceso tienen que preservar esas invariantes.

## Pedir prestado un libro

Ahora queremos una operación de dominio más rica:

```elixir
Library.lend_book(library, user_id, book_id)
```

La función tiene que verificar:

* que el usuario exista
* que el libro exista
* que el libro no esté ya prestado

Y si todo está bien:

* registrar el préstamo en `loans`
* actualizar el libro a `available: false`
* devolver una nueva biblioteca

---

## Implementación de `lend_book/3`

```elixir
def lend_book(%Library{} = library, user_id, book_id) do
  cond do
    not Map.has_key?(library.users, user_id) ->
      {:error, :user_not_found}

    not Map.has_key?(library.books, book_id) ->
      {:error, :book_not_found}

    Map.has_key?(library.loans, book_id) ->
      {:error, :already_lent}

    true ->
      book = Map.fetch!(library.books, book_id)
      updated_book = %Book{book | available: false}

      new_library = %Library{
        library
        | books: Map.put(library.books, book_id, updated_book),
          loans: Map.put(library.loans, book_id, user_id)
      }

      {:ok, new_library}
  end
end
```

## Cómo procesa mensajes el proceso

El proceso tiene una mailbox.

Los mensajes llegan ahí y el `receive` los atiende de a uno.

La idea general es:

1. llega un mensaje
2. el proceso lo pattern-matchea
3. aplica una transformación sobre el estado actual
4. obtiene una nueva versión del `%Library{}`
5. entra otra vez en `loop(new_state)`

Ese patrón es el corazón del modelo.

---

## Diagrama conceptual del loop

```text
mensaje 1 ---> mailbox ---┐
mensaje 2 ---> mailbox ---┼--> receive --> transformar estado --> loop(nuevo_estado)
mensaje 3 ---> mailbox ---┘
```

No hay dos transformaciones ejecutándose a la vez sobre la misma biblioteca.

Hay una secuencia de mensajes atendidos por el mismo proceso.

## Cómo se evitan las race conditions

El punto importante no es solo que usemos mensajes.

El punto importante es que:

* el `%Library{}` vive dentro de un único proceso
* ese proceso es el único que puede transformarlo
* los mensajes se procesan secuencialmente
* cada operación usa el estado más reciente

Entonces, si dos usuarios intentan pedir el mismo libro al mismo tiempo, no ocurre esto:

* ambos leen la misma estructura compartida
* ambos la modifican a la vez
* ambos escriben resultados incompatibles

Ocurre esto:

1. un mensaje entra primero
2. se procesa y actualiza `loans`
3. el siguiente mensaje se evalúa sobre el estado nuevo
4. si el libro ya está prestado, falla con `{:error, :already_lent}`

---

## Diagrama: dos usuarios piden el mismo libro

```text
Usuario A ---- {:lend_book, 1, 2, caller_a} ---->
                                               mailbox ---> LibraryServer
Usuario B ---- {:lend_book, 2, 2, caller_b} ---->
```

Procesamiento conceptual:

```text
Estado inicial:
book 2 available = true
loans = %{}

1. se atiende mensaje de A
2. se actualiza estado:
   - book 2 available = false
   - loans = %{2 => 1}
3. se atiende mensaje de B
4. B ve el nuevo estado
5. respuesta: {:error, :already_lent}
```

---

## ¿Por qué no hacen falta locks?

Porque no hay memoria compartida mutable entre procesos.

Cada proceso BEAM:

* tiene su propia memoria
* no puede modificar directamente la memoria de otro proceso
* se comunica por mensajes

Entonces no estamos en el modelo clásico de:

* varios threads
* mismo objeto compartido
* locks para protegerlo

Estamos en este otro modelo:

* estado aislado
* dueño único del estado
* acceso secuencial vía mailbox

---

## ⚠️ Trade-off: este modelo también tiene costos

Tener un único proceso dueño del estado simplifica mucho la consistencia:

* evita shared mutable state
* evita locks explícitos
* reduce race conditions
* serializa operaciones sobre ese recurso

Pero justamente por eso también introduce un costo:

* todas las operaciones pasan por la misma mailbox
* el proceso atiende una por vez
* puede convertirse en un cuello de botella

---

## Dónde aparece el cuello de botella

En nuestro ejemplo:

* muchos usuarios pueden mandar mensajes al mismo tiempo
* pero `LibraryServer` procesa secuencialmente
* la concurrencia existe del lado de los clientes
* el acceso al estado se serializa en el servidor

Eso evita incoherencias, pero limita el paralelismo sobre ese recurso.

---

## Problemas concretos que puede traer

### 1. Mailbox creciendo

Si los mensajes llegan más rápido de lo que el proceso puede atenderlos:

* la mailbox crece
* aumenta la latencia
* el proceso empieza a atrasarse

### 2. Un solo punto caliente

Aunque la VM tenga muchos schedulers, este proceso sigue ejecutando una sola operación por vez.

### 3. Proceso demasiado grande

Si le damos demasiadas responsabilidades:

* libros
* usuarios
* préstamos
* búsquedas
* reportes
* lógica pesada

terminamos con un proceso lento y difícil de escalar.

### 4. Un único punto de falla lógica

Si todo pasa por el mismo proceso, cualquier bug o lentitud ahí afecta a toda esa unidad del sistema.

---

## ¿Cómo se resuelve esto?

No agregando locks.

Se resuelve diseñando mejor la granularidad del estado y de los procesos.

---

## Estrategia 1: particionar el estado

En vez de tener un solo proceso para toda la biblioteca, podríamos tener:

* un proceso coordinador
* un proceso por libro
* un proceso por usuario
* o procesos por subconjuntos del dominio

Así reducimos contención y ganamos más paralelismo real.

---

## Estrategia 2: separar lectura de escritura

Muchas veces:

* escribir requiere serialización
* leer puede escalar con otros mecanismos

En sistemas más grandes, lecturas y escrituras no siempre tienen que pasar por el mismo camino.

---

## Estrategia 3: no meter trabajo pesado dentro del proceso dueño del estado

El proceso dueño del estado no debería hacer:

* parsing pesado
* IO pesado
* llamadas externas lentas
* cálculos largos

Porque mientras hace eso, no atiende su mailbox.

La regla práctica es:

> el proceso dueño del estado debería decidir rápido y delegar trabajo pesado.

---

## Idea de diseño importante

Un proceso dueño del estado no es la solución final a toda concurrencia.

Es una **unidad de consistencia**.

Después hay que decidir cuán grande o chica debe ser esa unidad.

---

## Comparación conceptual con objetos

La comparación con objetos puede servir como intuición, pero es solo ilustrativa.

### Similitudes

* ambos representan una entidad del dominio
* ambos preservan invariantes
* ambos exponen operaciones

### Diferencias

* el objeto suele vivir en memoria compartida
* el proceso vive aislado
* al objeto se lo invoca directamente
* al proceso se le mandan mensajes
* el objeto puede requerir locks si hay threads
* el proceso evita shared mutable state

El proceso no es “un objeto con otro nombre”.

Es una unidad de concurrencia.

---

## Qué queremos mostrar con este ejemplo

### 1. El proceso mantiene una estructura real

No solo un contador.

### 2. Cada mensaje puede:

* consultar
* agregar
* sacar
* transformar

### 3. El loop recursivo mantiene viva la abstracción

### 4. El estado sigue siendo inmutable

Cada operación construye una nueva versión de `%Library{}`.

### 5. La consistencia aparece por propiedad del estado

No por locks sobre memoria compartida.

## Cierre

Hoy vimos que un proceso puede administrar una abstracción de dominio completa.

La idea importante no es solo:

* crear procesos
* mandar mensajes

La idea importante es esta:

> un proceso puede ser el dueño de un estado estructurado, y cada mensaje puede transformar ese estado en una nueva versión.

Eso es el puente real entre:

* programación funcional con datos inmutables
* concurrencia basada en procesos
