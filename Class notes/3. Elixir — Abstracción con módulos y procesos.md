# Elixir — Abstracción con módulos y procesos

**Clase 2 — Del modelado de dominio al estado vivo en un proceso**

---

## 🎯 Objetivo

Entender cómo construir una abstracción real en Elixir usando:

- módulos
- maps y structs
- actualización inmutable de datos jerárquicos

Y luego dar el siguiente paso:

- mover esa abstracción a un proceso
- comunicarse por mensajes
- mantener estado dentro de un loop

---

## 💣 Idea central de la clase

Primero modelamos una abstracción real con funciones puras.

Después hacemos que esa abstracción viva dentro de un proceso.

---

## 🔁 De dónde venimos

Hasta ahora vimos:

- tipos de datos
- pattern matching
- control de flujo
- recursión
- pipelines
- `Enum`

Ahora vamos a usar esas herramientas para construir algo más grande.

---

## 🧠 Tesis de la clase

En Elixir:

- una abstracción se construye con datos + módulos
- cuando hay concurrencia, ese estado puede vivir dentro de un proceso
- el acceso al estado no se hace por memoria compartida
- se hace por paso de mensajes

Vamos a modelar una tienda online simple.

Queremos representar:

- productos

- operaciones sobre la tienda

Y después vamos a hacer que esa tienda viva dentro de un proceso.

### `Product`

### `Cart`

- `id`
- `items`

### `Shop`

- productos

## 🔧 Primer paso: structs del dominio

```elixir
defmodule Product do
  defstruct [:id, :name, :price, :stock]
end

defmodule Cart do
  defstruct [:id, items: %{}]
end

defmodule Shop do
  defstruct next_product_id: 1,
            next_cart_id: 1,
            products: %{},
            carts: %{}
end
```

### 🧠 Qué se ve acá

- el dominio está explícito
- cada struct representa una entidad
- no hay clases ni objetos
- siguen siendo datos inmutables

### 🔧 La abstracción vive en el módulo

En Elixir, la lógica no vive “dentro del objeto”. Vive en funciones de módulo.

```elixir
Shop.add_product(shop, product_data)
Shop.new_cart(shop)
Shop.add_to_cart(shop, cart_id, product_id, quantity)
```

### ⚠️ Esto no es OO clásico

No hacemos esto:

```elixir
shop.add_product(...)
cart.add_item(...)
```

Hacemos esto:

```elixir
Shop.add_product(shop, ...)
Shop.add_to_cart(shop, ...)
```

La abstracción está en:

- la representación de datos
- la API del módulo

## 📦 Agregar un producto

Queremos algo así:

```elixir
shop =
  Shop.new()
  |> Shop.add_product(%{name: "Mouse", price: 20000, stock: 10})
```

### 🔧 Implementación de `add_product/2`

```elixir
def add_product(%Shop{} = shop, attrs) do
  product = %Product{
    id: shop.next_product_id,
    name: attrs.name,
    price: attrs.price,
    stock: attrs.stock
  }

  %Shop{
    shop
    | next_product_id: shop.next_product_id + 1,
      products: Map.put(shop.products, product.id, product)
  }
end
```

### 🔍 Qué se ve en este ejemplo

- pattern matching en argumentos
- construcción de structs
- actualización inmutable
- generación de IDs
- uso de `Map.put/3`

## 🛒 Crear un carrito

```elixir
def new_cart(%Shop{} = shop) do

  new_shop = %Shop{
```

## 🛒 Agregar un producto al carrito

Queremos una función:

```elixir
Shop.add_to_cart(shop, cart_id, product_id, quantity)
```

Eso implica:

- buscar el carrito
- actualizar sus items
- reconstruir el carrito
- reconstruir la tienda

### 🔧 Implementación de `add_to_cart/4`

```elixir
def add_to_cart(%Shop{} = shop, cart_id, product_id, quantity) do
  cart = Map.fetch!(shop.carts, cart_id)

  new_items =
    Map.update(cart.items, product_id, quantity, fn old_qty ->
      old_qty + quantity
    end)

  new_cart = %Cart{cart | items: new_items}

  %Shop{shop | carts: Map.put(shop.carts, cart_id, new_cart)}
end
```

---

## 💰 Total de un carrito

```elixir
def cart_total(%Shop{} = shop, cart_id) do
  cart = Map.fetch!(shop.carts, cart_id)

  cart.items
  |> Enum.map(fn {product_id, quantity} ->
    product = Map.fetch!(shop.products, product_id)
    product.price * quantity
  end)
  |> Enum.sum()
end
```

## ✅ Checkout

Ahora queremos procesar una compra. Eso significa:

- verificar stock
- descontar stock de cada producto
- eliminar el carrito
- devolver una nueva tienda

### ⚠️ Simplificación de la clase

Vamos a asumir:

- stock suficiente
- sin pagos
- sin usuarios
- sin persistencia

Nos importa el modelo de estado y la transición de estado.

### 🔧 Implementación de `checkout/2`

```elixir
def checkout(%Shop{} = shop, cart_id) do
  cart = Map.fetch!(shop.carts, cart_id)

  new_products =
    Enum.reduce(cart.items, shop.products, fn {product_id, quantity}, products_acc ->
      product = Map.fetch!(products_acc, product_id)
      updated_product = %Product{product | stock: product.stock - quantity}
      Map.put(products_acc, product_id, updated_product)
    end)

  new_carts = Map.delete(shop.carts, cart_id)

  %Shop{shop | products: new_products, carts: new_carts}
end
```

## 🧠 Punto clave de esta primera parte

Hasta acá construimos una abstracción real en Elixir con:

- structs
- maps
- funciones de módulo
- actualizaciones inmutables

Todo esto funciona muy bien. Pero todavía hay un problema.

---

## ⚠️ ¿Qué problema aparece cuando hay varios clientes?

Mientras `Shop` es solo un dato puro, el modelo es claro y seguro. Pero cuando varias partes del sistema quieren operar “al mismo tiempo” sobre ese estado, aparece un problema clásico:

- dos clientes pueden leer la misma versión del estado
- ambos pueden decidir que una operación es válida
- ambos pueden intentar escribir una nueva versión
- una operación puede pisar a la otra

Eso es una **race condition**.

---

## 💣 Ejemplo conceptual de race condition

Supongamos este producto:

```elixir
%Product{id: 1, name: "Mouse", stock: 1, price: 20000}
```

Y dos clientes intentan comprarlo al mismo tiempo.

Los dos leen:

```elixir
stock = 1
```

Entonces ambos concluyen:

- “hay stock suficiente”
- “puedo vender una unidad”

Si ambos actualizan el estado por separado, el sistema puede terminar aceptando dos compras para una sola unidad disponible.

---

## 🧠 ¿Qué es una race condition?

Una race condition ocurre cuando:

- el resultado depende del orden exacto en que se ejecutan operaciones concurrentes
- varias operaciones acceden al mismo estado lógico
- no hay una coordinación clara entre esas operaciones

El problema no es solo “que haya muchas cosas pasando”. El problema es que varias operaciones compiten por decidir el próximo estado.

---

## 🔒 ¿Cómo suele resolverse esto en otros modelos?

En modelos con memoria compartida, el enfoque clásico es:

- varios threads acceden al mismo objeto o estructura
- como todos comparten memoria, hay que coordinar el acceso
- para eso se usan locks, mutexes, `synchronized`, etc.

La idea es: solo una ejecución entra a la vez a la sección crítica.

---

## ☕ Comparación conceptual con Java

En un modelo típico con objetos y threads, uno podría tener algo así:

```java
class Inventory {
    private int stock;

    public synchronized boolean buyOne() {
        if (stock > 0) {
            stock = stock - 1;
            return true;
        }
        return false;
    }
}
```

La palabra `synchronized` aparece porque:

- varios threads podrían entrar al mismo tiempo
- todos comparten el mismo objeto
- todos comparten la misma memoria
- entonces hay que bloquear acceso concurrente

---

## 🔁 En Elixir el enfoque es distinto

En Elixir no resolvemos esto compartiendo memoria y agregando locks. Lo resolvemos así:

- el estado vive dentro de un proceso
- ese proceso es el único dueño del estado
- otros procesos no tocan ese estado directamente
- solo pueden pedir operaciones enviando mensajes

Entonces el problema cambia completamente.

Ya no preguntamos:

> ¿cómo hago para que dos threads no toquen la misma memoria al mismo tiempo?

Preguntamos:

> ¿qué proceso es dueño del estado, y qué mensajes acepta?

---

## ✅ ¿Por qué no hacen falta locks?

Porque no hay memoria compartida mutable entre procesos.

Cada proceso BEAM:

- tiene su propia memoria
- no puede modificar directamente la memoria de otro proceso
- se comunica solamente por mensajes

Entonces no existe este problema:

> dos threads tocando simultáneamente la misma estructura mutable

El proceso dueño del estado recibe mensajes uno por uno y decide cómo transformar su estado.

---

## 🧠 Idea clave

Los locks aparecen cuando varias entidades comparten memoria mutable.

En Elixir, los procesos no comparten memoria mutable.

Por eso el modelo natural no es:

> shared state + locks

sino:

> isolated state + message passing

---

## 🔍 Proceso con estado vs objeto con estado

La comparación con objetos puede servir como intuición, pero es solo ilustrativa.

Un proceso con estado se parece a un objeto con estado en algo importante:

- ambos “mantienen” una forma de estado interno
- ambos exponen una interfaz de operaciones
- ambos permiten pensar en una entidad que responde pedidos

Pero la diferencia real es grande.

### 🔍 Similitudes ilustrativas

Un objeto y un proceso con estado pueden parecerse en que:

- reciben pedidos
- encapsulan cierta lógica
- preservan invariantes del dominio
- representan una entidad viva del sistema

Por ejemplo, `ShopServer` puede verse superficialmente como una entidad que “administra la tienda”.

### ❌ Diferencias importantes

**Objeto en OO clásico**

- suele vivir en memoria compartida
- varios threads pueden intentar usarlo al mismo tiempo
- necesita mecanismos de sincronización si hay concurrencia
- sus métodos se invocan directamente

**Proceso en Elixir**

- vive aislado, con memoria propia
- otros procesos no pueden tocar su estado interno
- se interactúa enviando mensajes
- procesa operaciones secuencialmente desde su mailbox

### 🧠 Diferencia central

Un objeto suele estar “ahí” para ser llamado directamente.

Un proceso en Elixir es una entidad concurrente autónoma que:

- recibe mensajes
- decide cómo responder
- mantiene su estado en su propio loop

No es solo “un objeto pero en otro lenguaje”. Es una unidad de concurrencia.

---

## 💣 El problema, ahora visto distinto

Ya no queremos una estructura pura que cualquiera reconstruya por su cuenta.

Queremos una única entidad que custodie el estado actual de la tienda.

Ahí aparece el proceso.

---

## 🔀 Paso a procesos

Ahora vamos a hacer que la tienda viva dentro de un proceso.

Ese proceso será:

- dueño del estado
- punto único de acceso
- quien decide cómo aplicar cada operación

---

## 🧠 Qué es un proceso en BEAM

Un proceso BEAM:

- no es un proceso del sistema operativo
- es muy liviano
- tiene su propia memoria
- tiene una mailbox
- se comunica por mensajes

### ⚠️ Proceso BEAM ≠ proceso del sistema operativo

Cuando decimos “proceso” en Elixir no hablamos de:

- un proceso pesado del OS
- memoria compartida
- threads clásicos

Hablamos de una unidad liviana de concurrencia de la VM.

---

## ✉️ Crear un proceso

```elixir
pid =
  spawn(fn ->
    IO.puts("Hola desde otro proceso")
  end)
```

---

## ✉️ Enviar mensajes

```elixir
pid =
  spawn(fn ->
    receive do
      message -> IO.inspect(message)
    end
  end)

send(pid, {:hello, "world"})
```

---

## 🧠 `receive`

`receive`:

- espera mensajes en la mailbox
- intenta matchear patrones
- si no encuentra uno, sigue esperando

---

## 🕓 `receive` con timeout

```elixir
receive do
  msg -> IO.inspect(msg)
after
  10000 -> IO.puts("timeout")
end
```

---

## 🧠 Request / response manual

Si queremos pedirle algo a un proceso y esperar respuesta, necesitamos:

- enviarle el mensaje
- incluir quién es el caller
- que el proceso responda a ese caller

### 🔧 Ejemplo simple

```elixir
server =
  spawn(fn ->
    receive do
      {:ping, caller} ->
        send(caller, :pong)
    end
  end)

send(server, {:ping, self()})

receive do
  msg -> IO.inspect(msg)
end
```

---

## 🔁 Un proceso útil necesita loop

Si un proceso hace un solo `receive`, maneja un mensaje y termina.

Para que pueda seguir vivo, necesitamos un loop recursivo.

```elixir
def loop do
  receive do
    msg ->
      IO.inspect(msg)
      loop()
  end
end
```

### 🧠 Acá vuelve recursión

Los procesos en Elixir viven gracias a un loop recursivo. Eso conecta perfecto con la clase anterior.

## 🏪 `ShopServer`: mover la tienda a un proceso

Ahora queremos que un proceso mantenga internamente un `%Shop{}`.

La idea es:

- arrancar el proceso con un estado inicial
- recibir mensajes
- actualizar estado
- seguir con un nuevo loop

### 🔧 Esqueleto del servidor

```elixir
defmodule ShopServer do
  def start do
    spawn(fn -> loop(Shop.new()) end)
  end

  def loop(shop) do
    receive do
      _message ->
        loop(shop)
    end
  end
end
```

### 🧠 Qué cambia respecto a `Shop`

Antes:

- `Shop` era una abstracción pura
- el caller manejaba el estado

Ahora:

- el estado vive dentro del proceso
- el caller no lo toca directamente
- solo manda mensajes

---

## ➕ Soportar `add_product`

```elixir
def loop(shop) do
  receive do
    {:add_product, attrs} ->
      new_shop = Shop.add_product(shop, attrs)
      loop(new_shop)
  end
end
```

---

## 👀 Consultar un producto

Queremos pedir:

```elixir
{:get_product, product_id, caller}
```

Y responder:

```elixir
{:product, product}
```

### 🔧 Ejemplo

```elixir
def loop(shop) do
  receive do
    {:get_product, product_id, caller} ->
      product = Map.get(shop.products, product_id)
      send(caller, {:product, product})
      loop(shop)

    {:add_product, attrs} ->
      new_shop = Shop.add_product(shop, attrs)
      loop(new_shop)
  end
end
```

## 🛒 Agregar al carrito desde el servidor

```elixir
{:add_to_cart, cart_id, product_id, quantity}
```

Ese mensaje:

- modifica el estado
- no necesita respuesta inmediata en esta versión simple

## 🧾 Crear un carrito desde el servidor

Acá sí necesitamos responder, porque el caller necesita el `cart_id`.

```elixir
{:new_cart, caller}
```

### 🔧 Ejemplo más completo

```elixir
def loop(shop) do
  receive do
    {:new_cart, caller} ->
      {new_shop, cart_id} = Shop.new_cart(shop)
      send(caller, {:cart_created, cart_id})
      loop(new_shop)

    {:add_product, attrs} ->
      new_shop = Shop.add_product(shop, attrs)
      loop(new_shop)

    {:get_product, product_id, caller} ->
      product = Map.get(shop.products, product_id)
      send(caller, {:product, product})
      loop(shop)

    {:add_to_cart, cart_id, product_id, quantity} ->
      new_shop = Shop.add_to_cart(shop, cart_id, product_id, quantity)
      loop(new_shop)

    {:checkout, cart_id} ->
      new_shop = Shop.checkout(shop, cart_id)
      loop(new_shop)
  end
end
```

---

## ✅ Aplicado a `ShopServer`

Cuando hacemos que `%Shop{}` viva dentro de `ShopServer`:

- solo `ShopServer` puede decidir el próximo estado
- nadie más modifica directamente `products` o `carts`
- todas las operaciones llegan como mensajes
- el loop las procesa de a una

Eso reduce muchísimo el espacio donde pueden aparecer race conditions.

---

## 💣 Ejemplo conceptual: compra concurrente

Si dos clientes mandan al mismo tiempo:

```elixir
{:checkout, cart_id_1}
{:checkout, cart_id_2}
```

el proceso no ejecuta ambas operaciones “en paralelo” sobre el mismo estado mutable. Hace esto:

- recibe un mensaje
- actualiza el estado
- sigue con el siguiente mensaje
- aplica la segunda operación sobre el nuevo estado

Eso significa que la segunda operación ya ve el estado actualizado por la primera.

---

## 🧠 Consecuencia

El proceso actúa como un serializador natural de operaciones sobre ese estado.

No hace falta lockear cada campo o estructura.

La consistencia aparece porque:

- hay un único dueño del estado
- el acceso es secuencial
- el estado no se comparte

---

## 🧠 Qué ganamos al usar un proceso

Ahora tenemos:

- un único dueño del estado
- operaciones serializadas
- una forma consistente de aplicar cambios
- paso de mensajes en lugar de memoria compartida

### ⚠️ Punto importante

Un proceso procesa mensajes secuencialmente. Eso significa:

- no ejecuta dos operaciones al mismo tiempo
- atiende una, actualiza estado, sigue con la próxima

Eso ayuda a mantener consistencia.

---

## 🏷️ Registrar un proceso

En vez de guardar el PID, podemos registrar el proceso con nombre.

```elixir
pid = spawn(fn -> ShopServer.loop(Shop.new()) end)
Process.register(pid, :shop_server)
```

Y después:

```elixir
send(:shop_server, {:add_product, %{name: "Mouse", price: 20000, stock: 10}})
```
