# Elixir — Fundamentos: datos, pattern matching, control de flujo, recursión y pipelines

Clase 1 — Bases del lenguaje antes de concurrencia

## 🎯 Objetivo

Antes de ver concurrencia es necesario entender:

- cómo se representan los datos
- cómo fluye el código
- cómo se transforman los datos

## 💣 Cambio de paradigma

En Elixir:

- no hay estado mutable
- no hay loops clásicos
- no hay objetos
- no hay memoria compartida

Programar consiste en transformar datos.

## 🧱 Tipos de datos

### Tuplas

```elixir
{:ok, 42}
{:error, :not_found}
```

- tamaño fijo
- acceso rápido
- muy usadas para representar resultados

### Listas

```elixir
[1, 2, 3]
["hola", "mundo"]
[:a, :b, :c]
```

- listas enlazadas
- acceso secuencial `O(n)`
- construir al frente es `O(1)`

```elixir
[h | t] = [1, 2, 3]
# h = 1
# t = [2, 3]
```

### Maps

```elixir
%{name: "Juan", age: 30}
```

- estructura clave → valor
- dinámicos
- muy útiles para modelar datos tipo JSON

### Keyword lists

```elixir
[name: "Juan", age: 30]
```

- lista de tuplas
- mantiene orden
- permite claves repetidas

```elixir
[mode: :fast, debug: true, mode: :safe]
```

## 🔥 Pattern Matching

El operador `=` no asigna: intenta hacer match entre dos expresiones.

### Caso básico

```elixir
a = 1
```

### Matching entre expresiones

```elixir
1 = 1
```

```elixir
1 = 2
# error
```

### Rebinding

```elixir
a = 1
a = 2
```

Las variables no cambian de valor: se crea un nuevo binding.

### Tuplas

```elixir
{a, b} = {1, 2}
{:ok, value} = {:ok, 42}
```

### Listas

```elixir
[h | t] = [1, 2, 3]
[first, second] = [10, 20]
[first, second] = [10, 20, 30]
# error
```

### Maps

```elixir
%{name: name} = %{name: "Juan", age: 30}
%{name: name, age: age} = %{name: "Juan", age: 30}
```

En maps no hace falta matchear todas las claves.

### Matching anidado

```elixir
{:ok, %{name: name}} = {:ok, %{name: "Juan"}}
%{user: %{name: name}} = %{user: %{name: "Ana", age: 20}}
```

### Ignorar valores

```elixir
{_, b} = {1, 2}
[h | _] = [1, 2, 3]
```

### Pin operator (`^`)

Sirve para matchear contra un valor ya existente.

```elixir
a = 1
^a = 1
```

```elixir
a = 1
^a = 2
# error
```

### En definiciones de funciones

```elixir
def classify(0), do: :zero
def classify(1), do: :one
def classify(_), do: :other

def greet(%{name: name}) do
  "Hola #{name}"
end

def extract_ok({:ok, value}) do
  value
end
```

El comportamiento puede definirse según la forma de los datos.

### Guards

Permiten agregar condiciones a una cláusula.

```elixir
def is_adult(age) when age >= 18, do: true
def is_adult(_), do: false

def sign(x) when x > 0, do: :positive
def sign(0), do: :zero
def sign(x) when x < 0, do: :negative
```

## 🧊 Inmutabilidad

```elixir
x = 1
x = x + 1
```

Las variables no cambian de valor: se redefinen.

### ⚠️ Consecuencias

- no hay estado compartido
- no hay efectos secundarios implícitos
- el código es más predecible
- es mucho más fácil razonar sobre lo que hace un programa

## 🔀 Control de flujo

En Elixir, el control de flujo se apoya mucho en pattern matching.

### `case`

Se usa cuando el flujo depende de la forma del dato.

```elixir
case x do
  0 -> :zero
  1 -> :one
  _ -> :other
end
```

### `case` con estructuras

```elixir
case result do
  {:ok, value} -> value
  {:error, _reason} -> :error
end

case user do
  %{name: name, age: age} when age >= 18 -> "#{name} es mayor"
  %{name: name} -> "#{name} es menor"
end
```

### `cond`

Se usa para múltiples condiciones booleanas.

```elixir
cond do
  x > 10 -> :big
  x > 5 -> :medium
  true -> :small
end
```

### `if`

Se usa para casos simples.

```elixir
if x > 0 do
  :positive
else
  :negative
end
```

### ⚠️ Recomendaciones

- preferir pattern matching cuando sea posible
- usar `case` cuando el flujo depende de estructuras
- usar `cond` para varias condiciones booleanas
- usar `if` solo cuando la lógica sea simple

## 🔁 Recursión

No existen loops tradicionales.

La iteración se expresa mediante recursión.

### Forma general

Toda función recursiva tiene:

- caso base
- caso recursivo

### Ejemplo — suma

```elixir
def sum([]), do: 0
def sum([h | t]), do: h + sum(t)
```

### Ejemplo — length

```elixir
def length([]), do: 0
def length([_ | t]), do: 1 + length(t)
```

### Ejemplo — map recursivo

```elixir
def map([], _fun), do: []

def map([h | t], fun) do
  [fun.(h) | map(t, fun)]
end
```

### Ejemplo — filter recursivo

```elixir
def filter([], _fun), do: []

def filter([h | t], fun) do
  if fun.(h) do
    [h | filter(t, fun)]
  else
    filter(t, fun)
  end
end
```

### Ejemplo — contar pares

```elixir
def count_even([]), do: 0

def count_even([h | t]) when rem(h, 2) == 0 do
  1 + count_even(t)
end

def count_even([_ | t]) do
  count_even(t)
end
```

### Ejemplo — sumar pares de una lista

```elixir
def sum_pairs([]), do: 0

def sum_pairs([{a, b} | t]) do
  a + b + sum_pairs(t)
end
```

### Ejemplo — contar adultos en una lista de maps

```elixir
def count_adults([]), do: 0

def count_adults([%{age: age} | t]) when age >= 18 do
  1 + count_adults(t)
end

def count_adults([_ | t]) do
  count_adults(t)
end
```

## ⚙️ Tail Recursion

La recursión de cola es una forma especial de recursión que la VM puede optimizar.

### 🔁 Recursión normal

```elixir
def sum([]), do: 0
def sum([h | t]), do: h + sum(t)
```

### Qué pasa acá

Si ejecutamos:

```elixir
sum([1, 2, 3])
```

la evaluación queda conceptualmente así:

```elixir
1 + sum([2, 3])
1 + (2 + sum([3]))
1 + (2 + (3 + sum([])))
1 + (2 + (3 + 0))
```

Cada llamada deja trabajo pendiente.

### ⚠️ Consecuencia

- cada llamada necesita recordar qué hacer después
- eso usa stack
- el stack crece con el tamaño del problema

### Tail recursion

Una función es tail recursive cuando la última operación es la llamada recursiva.

```elixir
def sum(list), do: sum(list, 0)

def sum([], acc), do: acc
def sum([h | t], acc), do: sum(t, acc + h)
```

### 🧠 Qué cambia

Ahora la evaluación queda así:

```elixir
sum([1, 2, 3], 0)
sum([2, 3], 1)
sum([3], 3)
sum([], 6)
```

No queda trabajo pendiente después de la llamada recursiva.

### 🧠 Manejo de memoria

En la recursión normal:

- cada llamada agrega un frame al stack
- luego hay que volver hacia atrás resolviendo operaciones pendientes

En tail recursion:

- como no queda trabajo pendiente, la VM puede reutilizar el mismo frame
- el uso de memoria se mantiene constante
- el comportamiento se parece al de un loop eficiente

### ⚠️ Regla práctica

Si aparece algo así:

```elixir
h + recursion(...)
```

no es tail recursive.

Si aparece algo así:

```elixir
recursion(..., acc + h)
```

sí es tail recursive.

### Ejemplo — reverse no optimizado

```elixir
def reverse([]), do: []

def reverse([h | t]) do
  reverse(t) ++ [h]
end
```

- no es tail recursive
- `++` es costoso
- el stack crece

### Ejemplo — reverse optimizado

```elixir
def reverse(list), do: reverse(list, [])

defp reverse([], acc), do: acc
defp reverse([h | t], acc), do: reverse(t, [h | acc])
```

### Evaluación

```elixir
reverse([1, 2, 3], [])
reverse([2, 3], [1])
reverse([3], [2, 1])
reverse([], [3, 2, 1])
```

### Ejemplo — length optimizado

```elixir
def length(list), do: length(list, 0)

defp length([], acc), do: acc
defp length([_ | t], acc), do: length(t, acc + 1)
```

### Ejemplo — map optimizado

```elixir
def map(list, fun), do: map(list, fun, [])

defp map([], _fun, acc), do: Enum.reverse(acc)

defp map([h | t], fun, acc) do
  map(t, fun, [fun.(h) | acc])
end
```

## 🔢 Fibonacci

### Versión simple

```elixir
def fib(0), do: 0
def fib(1), do: 1
def fib(n) when n > 1, do: fib(n - 1) + fib(n - 2)
```
### Evaluacion:
```
fib(5)
fib(4) + fib(3)
(fib(3) + fib(2)) + (fib(2) + fib(1))
((fib(2) + fib(1)) + (fib(1) + fib(0))) + ((fib(1) + fib(0)) + fib(1))
(((fib(1) + fib(0)) + fib(1)) + (fib(1) + fib(0))) + ((fib(1) + fib(0)) + fib(1))
((1 + 0) + 1) + (1 + 0) + (1 + 0) + 1
5
```

Esta versión es clara, pero recalcula muchísimo y no es tail recursive.

### 🔍 Qué se ve acá

`fib(3)` se calcula más de una vez
`fib(2)` se calcula más de una vez
`fib(1)` y `fib(0)` aparecen muchas veces
cada llamada deja trabajo pendiente

### ⚠️ Consecuencia

#### Esta versión:

- usa stack
- no reutiliza resultados
- crece muy mal
- se vuelve rápidamente ineficiente

### Versión tail recursive

```elixir
def fib(0), do: 0
def fib(1), do: 1
def fib(n), do: fib(n, 0, 1)

defp fib(1, _a, b), do: b
defp fib(n, a, b), do: fib(n - 1, b, a + b)
```

### 🧠 Idea

En vez de recalcular, se pasa el estado como parámetros:

- `a` = término anterior
- `b` = término actual

### Evaluación

```elixir
fib(5, 0, 1)
fib(4, 1, 1)
fib(3, 1, 2)
fib(2, 2, 3)
fib(1, 3, 5)
```

Resultado:

```elixir
5
```
### ⚠️ Consecuencia
- avanza en línea
- no recalcula
- lleva el estado como parámetros
- no deja trabajo pendiente
- el compilador la optimiza como tail recursive !!!

## 📚 Enums: De recursión explícita a abstracciones de recorrido

Hasta ahora vimos cómo recorrer listas “a mano” usando recursión.

Por ejemplo:

```elixir
def sum([]), do: 0
def sum([h | t]), do: h + sum(t)
```
o incluso:

```elixir
def map([], _fun), do: []

def map([h | t], fun) do
  [fun.(h) | map(t, fun)]
end
```

Esto es importante porque muestra cómo funciona realmente el lenguaje.

Pero en código real, muchas veces no queremos escribir esa recursión explícitamente una y otra vez.

Para eso existe Enum: un módulo que nos da abstracciones ya hechas para recorrer y transformar colecciones. Siendo sus funcione principales:

- `map` transforma cada elemento de una arreglo
- `filter` selecciona algunos elementos
- `reduce` acumula resultados

`reduce` es la abstracción más general.

## 🔥 Reduce

Puede expresar cualquier transformación sobre una colección. de hecho map y filter se peuden escribir usando solo reduce:

### map con reduce

```elixir
def map(list, fun) do
  list
  |> Enum.reduce([], fn x, acc ->
    [fun.(x) | acc]
  end)
  |> Enum.reverse()
end
```

### filter con reduce

```elixir
def filter(list, fun) do
  list
  |> Enum.reduce([], fn x, acc ->
    if fun.(x), do: [x | acc], else: acc
  end)
  |> Enum.reverse()
end
```

### 🧠 Idea

- `map` conserva cardinalidad
- `filter` cambia cardinalidad
- `reduce` permite construir ambos

## 🔗 Pipeline (`|>`): De composición anidada a lectura lineal

Con `Enum` ya tenemos herramientas para transformar colecciones, pero cuando combinamos varias operaciones aparece un problema de legibilidad.

Por ejemplo, esto es correcto:

```elixir
Enum.map(
  Enum.filter([1, 2, 3, 4], fn x -> x > 2 end),
  fn x -> x * 2 end
)
```
Sin embargo, el código queda anidado y se lee de adentro hacia afuera.

En muchos casos queremos expresar algo más cercano a esta idea:

- tomar una colección
- filtrar algunos elementos
- aplicar alguan funcion nuestra a toda una coleccion
- transformar el resultado
- seguir procesándolo paso a paso

El pipeline (|>) permite escribir justamente eso: una secuencia de transformaciones donde el resultado de un paso pasa al siguiente.


### 🧠 Idea básica

```elixir
value |> function()
```

equivale a:

```elixir
function(value)
```

### Ejemplo simple

```elixir
"hola"
|> String.upcase()
```

equivale a:

```elixir
String.upcase("hola")
```

### Encadenamiento

```elixir
"hola"
|> String.upcase()
|> String.reverse()
```

### Sin pipes
Antes de conocer este operador, esta operacion la realizabamos asi:

```elixir
String.reverse(String.upcase("hola"))
```

Es correcto, pero menos legible.

### 🧠 Regla clave

El valor de la izquierda se pasa como primer argumento de la función.

### Ejemplo con Enum

```elixir
[1, 2, 3, 4]
|> Enum.filter(fn x -> rem(x, 2) == 0 end)
|> Enum.map(fn x -> x * 2 end)
```

### Paso a paso

```elixir
Enum.filter([1, 2, 3, 4], fn x -> rem(x, 2) == 0 end)
Enum.map(resultado, fn x -> x * 2 end)
```

### El orden importa

```elixir
list
|> Enum.map(fn x -> x * 2 end)
|> Enum.filter(fn x -> x > 4 end)
```

no es lo mismo que:

```elixir
list
|> Enum.filter(fn x -> x > 4 end)
|> Enum.map(fn x -> x * 2 end)
```

### Con funciones propias

```elixir
def double(x), do: x * 2

[1, 2, 3]
|> Enum.map(&double/1)
```

### Múltiples argumentos

El pipe solo inserta el valor como primer argumento.

```elixir
Enum.map(list, fun)
```

permite escribir:

```elixir
list |> Enum.map(fun)
```

Si la función necesita más argumentos:

```elixir
list
|> Enum.map(fn x -> my_fun(x, extra) end)
```

### ⚠️ Recomendaciones

- usar pipes cuando hay varias transformaciones
- mantener cada paso simple
- evitar pipes cuando no mejoran legibilidad
- usar una función anónima si hace falta adaptar la forma de llamada

### Ejemplo poco claro

```elixir
list
|> Enum.map(fn x -> if x > 2 do x * 2 else x end end)
```

### Mejor versión

```elixir
list
|> Enum.filter(fn x -> x > 2 end)
|> Enum.map(fn x -> x * 2 end)
```

Elixir se apoya en:

- datos inmutables
- pattern matching
- control de flujo basado en estructuras
- recursión
- pipelines
- transformación de datos

Estas ideas son la base para entender después:

- procesos
- paso de mensajes
- GenServer
- OTP
